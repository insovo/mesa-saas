> 版本：v2（2026-05-27 迭代）  
> 适用项目：MESA Recruit  
> 模板来源：`/Users/mysaria/Project/mesa/属地员工面试评价表_标准单人版.xlsx`

## 1. 背景与目标

候选人详情页的右侧 aside 当前从上到下是：profile sticky → 面试安排 → 附件。HR 安排完面试后，缺一个让面试官把评分、纪要、结论沉淀到系统里的入口。线下用 Excel 模板填写虽然格式完整，但分散、易丢失、汇总难。

本模块在「面试安排」和「附件」之间插入一个「面试评价」卡片，整体能力：

- HR / 招聘官生成 **公开链接** → 面试官扫码或点链接，无需登录直接填写评价
- 在线表单交互（数字评分、实时加权计算、纪要文本框、评分标准抽屉参考）
- 一份评价可以多次保存草稿；提交后默认锁定（admin 可退回）
- 任一时刻可以一键导出 `.xlsx`，导出文件与原模板**完全一致**的版式（合并单元格 / 列宽 / 行高 / 公式 / 数据校验 / 冻结窗格全保留），仅把空白单元格填入本次评价内容
- 候选人详情页可看到该候选人所有评价邀请的状态（草稿 / 已提交 / 已撤销）、总分、推荐结论

设计原则：
1. **与现有体系对齐** — 复用 mesa 的 ShareLink/UploadShareLink token 设计、AuthGuard 外公开路由模式、AES 加密 SystemSetting、Cloudflare 100s 超时规避策略
2. **模板即源** — 字段映射、公式、校验全部源于 Excel 模板，模板换版本不破坏旧数据（templateVersion + fileHash 记录）
3. **最小可用先行** — V1 只覆盖「标准单人版」的端到端闭环，多面试官合并 / 多轮对比 / 批量评价 / 动态模板放 V2+

### 2.6 评分标准 Sheet（导出原样保留）

| 维度 | 9-10 分 | 7-8 分 | 5-6 分 | 1-4 分 |
| --- | --- | --- | --- | --- |
| 沟通表达 | 回答结构清晰，重点明确，能举出有效实例并回应追问 | 表达较清楚，偶有冗长，但能说明核心信息 | 表达基本能理解，但逻辑性和重点性一般 | 表达混乱，难以说明问题，追问后仍不清楚 |
| 岗位匹配度 | 核心经历高度贴合岗位，可快速上手并产出 | 大部分经历相关，经过短期磨合可胜任 | 部分相关，需要较多培训或适配 | 匹配度低，关键经历或能力明显不足 |
| 专业基础/业务理解 | 专业基础扎实，能结合案例说明做法与结果 | 基础较好，能覆盖岗位主要要求 | 基础一般，对关键知识理解不够深入 | 基础薄弱，回答停留在表面或明显错误 |
| 学习与适应能力 | 学习快，适应强，能主动补位并快速融入新场景 | 能较好适应变化，遇到新问题愿意学习 | 适应速度一般，需要更多引导 | 对变化抗拒明显，适应与学习意愿偏弱 |
| 责任心与执行力 | 目标意识强，能主动推进并对结果负责 | 执行较稳，能完成既定任务并及时反馈 | 执行基本可接受，但主动性和闭环意识一般 | 执行推动弱，责任边界不清，容易失约或失控 |
| 稳定性与求职动机 | 动机清晰真实，职业规划稳定，风险低 | 动机较明确，整体稳定性较好 | 动机一般，存在一定不确定性 | 动机不清、变动频繁或稳定性风险明显 |
| 文化契合与团队协作 | 合作意识强，尊重规则，能建设性处理分歧 | 团队协作较好，沟通开放 | 可以合作，但协作意识较弱 | 协作意愿弱，价值取向或行为方式存在明显风险 |

> 这张表在公开评价页将作为「评分标准」抽屉的内容来源（也可直接读 Sheet）。

## 4. 用户角色与核心场景

### 4.1 角色

| 角色 | 能力 |
| --- | --- |
| ADMIN | 全部能力 + 退回编辑 / 强制撤销 / 删除已提交记录 / 看跨候选人评价列表 |
| RECRUITER | 创建评价邀请 / 复制链接 / 查看本人创建的评价 / 导出 xlsx |
| 面试官（无登录，token 入站） | 填写表单 / 保存草稿 / 提交评价 / 导出本次评价 xlsx（按【决策点 4】） |

### 4.2 核心场景流（端到端）

1. 招聘官打开候选人详情，看到「面试评价」卡片为空，点击「+ 新建评价」
2. 选定面试轮次 + 面试官姓名 → 系统生成 `InterviewEvaluation` 记录，状态 `link_sent`，token 32 字符
3. 招聘官复制链接 / 截图二维码发给面试官
4. 面试官点链接打开 `/interview-eval/:token`，看到候选人姓名、岗位、面试时间已预填
5. 面试官打分 1-10 + 填写备注 + 写优势/风险/追问/最终意见 → 页面实时显示总分气泡和推荐结论
6. 面试官可点「保存草稿」(status = `draft`) 离开，后续再回来；或直接「提交评价」(status = `submitted`)
7. 提交后链接进入只读，按【决策点 2】决定是否允许面试官自助再改
8. 招聘官在候选人详情页看到状态变 `submitted`，点「⤓ 导出 xlsx」下载文件
9. 系统读模板 → 写入本次评价数据 → 流式输出，文件名 `属地员工面试评价表_姓名_岗位_面试日期.xlsx`

### 5.2 交互细节

- 评分用 **拖动进度条**（细步进丝滑拖动；落盘 / 导出仍为 **1–10 整数**）。拇指对齐、Codex 填充气泡、`onScrub` 喂总分球等完整规范见 [`评分拖条与填充气泡交互规范.md`](./评分拖条与填充气泡交互规范.md)
- 加权得分、总分、推荐结论**前端按相同公式实时算**（拖动中可用 float scrub 让 LiquidLoader 小数连续；与导出 xlsx 用模板公式时打开端算的结果一致）
- 「评分标准 ⓘ」按钮 → 抽屉打开，展示 §2.6 那张表（mobile 用 modal）
- 草稿每 30 秒自动 `PATCH /api/public/interview-eval/:token`（合并保存），切页前再保一次
- 提交前校验：7 项评分必填、最终意见必填，缺字段高亮 + 滚动到首个缺失项
- 移动端：评分项纵向 stack，备注 textarea 紧贴评分；action bar 固定底部
- 提交成功 → 页面切到只读视图，顶部 banner「评价已提交，感谢配合 ✓」+「下载本次评价 xlsx」按钮（按【决策点 4】控制可见）

### 5.3 公式注入防护（前端层）

任何用户输入的 textarea 如果以 `=`、`+`、`-`、`@` 开头，前端提交时**保留原样**（后端再做 sanitize），不在前端拦截，避免误伤合法内容如「-1 分原因…」。

| `D10:D16` | 权重值 | 保留模板原值，V1 不允许改 |
| `A29:G30` | 评分建议说明 | 保留模板原文 |
| `评分标准` 整 Sheet | 完整保留 |

> 前端展示用与公式同款 JS 实现：`weighted[i] = round(weight[i] * score[i] / 10, 1)`；`total = round(sum(weighted), 1)`；`recommend = total>=85?"建议录用":total>=75?"建议复试":total>=60?"谨慎考虑":"不建议录用"`。

## 7. 数据模型（Prisma）

按 mesa 现有规范：`@db.Uuid` + `@default(uuid())` + `@map("snake_case")` + `@@map("snake_case_table")`。

### 7.1 主表 `interview_evaluations`

```prisma
model InterviewEvaluation {
  id            String    @id @default(uuid()) @db.Uuid

  // 关联（评价归属候选人，可选关联具体 Interview 记录）
  candidateId   String    @map("candidate_id") @db.Uuid
  candidate     Candidate @relation(fields: [candidateId], references: [id], onDelete: Cascade)
  interviewId   String?   @map("interview_id") @db.Uuid
  interview     Interview? @relation(fields: [interviewId], references: [id], onDelete: SetNull)
  jobId         String?   @map("job_id") @db.Uuid
  job           Job?      @relation(fields: [jobId], references: [id], onDelete: SetNull)

  // 公开 token (32 字符 URL-safe random)
  token         String    @unique
  status        String    @default("link_sent")  // link_sent | draft | submitted | revoked | expired
  expiresAt     DateTime? @map("expires_at")

  // 候选人信息快照（V1 仅在创建时预填一次，面试官可改；后续 candidate 改名不回填）
  candidateName String    @map("candidate_name")
  position      String?
  region        String?
  interviewDate DateTime? @map("interview_date")
  interviewer   String                          // 面试官（必填，识别评价来源）
  languageStrength      String? @map("language_strength")
  currentCity           String? @map("current_city")
  department            String?
  timezoneCollaboration String? @map("timezone_collaboration")

  // 评分（7 维度数组快照，含 dimensionKey/score/remark；权重不存，从模板版本反查）
  // 形如:[{key:"communication",score:8,remark:"..."},...]
  scores        Json      @default("[]")

  // 纪要
  strengths         String? @db.Text
  risks             String? @db.Text
  followUpQuestions String? @db.Text @map("follow_up_questions")
  finalOpinion      String? @db.Text @map("final_opinion")

  // 汇总（提交时计算并存，导出时 Excel 公式重新算一遍兜底）
  totalScore     Float?  @map("total_score")
  recommendation String?                          // 建议录用 | 建议复试 | 谨慎考虑 | 不建议录用

  // 模板版本管理
  templateVersion  String  @default("v1") @map("template_version")
  templateFileHash String? @map("template_file_hash")  // SHA-256 of template xlsx, 落档时记录

  // 操作元信息
  createdBy       String?   @map("created_by") @db.Uuid    // 招聘官 user.id
  submittedAt     DateTime? @map("submitted_at")
  exportedAt      DateTime? @map("exported_at")
  exportedCount   Int       @default(0) @map("exported_count")
  lastViewedAt    DateTime? @map("last_viewed_at")
  viewCount       Int       @default(0) @map("view_count")
  revokedAt       DateTime? @map("revoked_at")
  revokedBy       String?   @map("revoked_by") @db.Uuid

  createdAt     DateTime  @default(now()) @map("created_at")
  updatedAt     DateTime  @updatedAt @map("updated_at")

  @@index([candidateId])
  @@index([interviewId])
  @@index([token])
  @@map("interview_evaluations")
}
```

### 7.2 设计要点

- **一份评价 = 一条记录**（不拆 link 表 + content 表，简化状态机；token 即 link）
- **多面试官 = 多条记录**（candidate × interviewer N:M，参见【决策点 1】）
- **状态机**：`link_sent`(初建) → `draft`(面试官保存过) → `submitted`(已提交) → `revoked`(撤销)；`expired` 由 expiresAt 决定，懒判定不入 DB
- **快照策略**：候选人信息预填后存进本表；候选人改名不回填评价快照（避免历史评价数据漂移）
- **评分明细**：V1 用 JSON 列 `scores`；如果未来需要按维度统计/筛选，再加 `interview_evaluation_scores` 明细表（一对多）
- **不存权重**：权重由 `templateVersion` 反查（代码里固化），方便未来调整权重时新模板版本独立追溯

### 7.3 在 `Candidate` 模型新增反向关系

```prisma
model Candidate {
  // ...
  interviewEvaluations InterviewEvaluation[]
}
```

`Interview` 和 `Job` 同步加反向。

### 7.4 迁移路径

`prisma/migrations/<timestamp>_add_interview_evaluations/migration.sql` 手写（兼容 mesa 现有迁移风格），含建表 + 加 FK + 加索引 + 不修改历史数据。

### 8.4 错误码（对齐 share.js）

| code | HTTP | 含义 |
| --- | --- | --- |
| `eval_not_found` | 404 | token 不存在 |
| `eval_expired` | 410 | expiresAt 已过 |
| `eval_revoked` | 410 | 被招聘官撤销 |
| `eval_already_submitted` | 409 | 已提交、且【决策点 2】不允许重复提交时 |
| `eval_validation_failed` | 422 | 必填缺失 / 评分超范围 |
| `eval_export_disabled` | 403 | 按【决策点 4】公开端不允许导出时 |

### 8.5 接口注意

- 公开接口**绝不返回** `createdBy` / `revokedBy` / 内部用户 ID
- 公开接口返回的 `interviewer` 即面试官姓名（评价人自己），可见 OK
- POST 提交做幂等：发现 status 已是 submitted 时直接返回当前记录，不报错
- 草稿 PATCH 用 deep merge：只更新传入的字段，避免分次保存覆盖
- 导出接口前置检查 `status === "submitted"`（防止草稿就导出）；admin 可强制导出当前状态

## 9. 导出方案（ExcelJS + 模板复制）

mesa 是 Node 后端，引入 `exceljs ^4.4.x`。模板文件入仓位置：

```
server/assets/templates/interview-evaluation-v1.xlsx
```

启动时计算文件 SHA-256 hash 缓存在内存，写入 `InterviewEvaluation.templateFileHash`。

### 9.1 导出流程

1. 读模板文件 → `new ExcelJS.Workbook().xlsx.load(fileBuffer)`
2. 取出 `面试评分表` worksheet
3. 按 §6 字段映射逐格写入（写值前 sanitize）
4. **不动**公式单元格 `F10:F17`, `G17`；**不动** `评分标准` Sheet；**不动**列宽 / 行高 / 合并 / 数据校验 / freeze
5. 文件名安全化：`属地员工面试评价表_${safe(candidateName)}_${safe(position)}_${yyyy-MM-dd}.xlsx`
6. `workbook.xlsx.writeBuffer()` → 流式回写 response，Content-Type `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`
7. 更新 `exportedAt` + `exportedCount++`

### 9.2 公式注入防护（后端层）

每个文本字段写入前过一遍 sanitize：

```js
function sanitizeForExcel(v) {
  if (v == null) return null;
  const s = String(v);
  // 开头是 = + - @ tab cr lf 视为危险,前缀单引号让 Excel 当纯文本
  if (/^[=+\-@\t\r\n]/.test(s)) return "'" + s;
  return s;
}
```

不对评分（int 1-10）做此处理。

### 9.3 文件名安全化

```js
function safeFilename(s) {
  return String(s || "未命名")
    .replace(/[\\/:*?"<>|\x00-\x1f]/g, "_")  // Windows / 控制字符
    .replace(/\.+$/, "")                      // 结尾点
    .slice(0, 80);                            // 长度截断
}
```

`Content-Disposition` 头同时给 RFC 5987 编码版本，兼容中文：

```
Content-Disposition: attachment; filename="fallback.xlsx"; filename*=UTF-8''%E5%B1%9E%E5%9C%B0...
```

### 9.4 公式重算

V1 **不在服务端 LibreOffice headless 重算**（理由见【决策点 5】）：

- 模板里 `F10:F17`、`G17` 公式已留全，ExcelJS 不删
- 用户用 Excel / WPS / LibreOffice / 在线 Excel 打开时自动计算
- 兜底：`InterviewEvaluation.totalScore` / `recommendation` 在提交时已算并存库，导出文件名 + 后续报表都可读 DB 而不依赖打开 xlsx

如有「打开前查看 F17 值」的强需求，V2 加 LibreOffice headless 步骤（docker image 多 ~400MB）。

### 9.5 ExcelJS 已知限制

- ExcelJS 加载老 xlsx 可能丢失部分**条件格式**和**主题颜色**（mesa 模板未用，影响小）
- 数据校验 ExcelJS 支持 `dataValidations`，理论保留；加载/保存往返必须用 Excel 实际验证一次
- 不支持嵌入图片重写（模板无图片，OK）

落地前必须做的回归：用 ExcelJS load → save 一次（不改任何数据），用 Excel 打开对比原模板视觉是否一致。

## 10. 权限与安全

### 10.1 Token 安全

- token = `crypto.randomBytes(24).toString("base64url")`（32 字符 URL-safe，沿用 ShareLink 设计）
- 默认有效期 7 天（最长 30 天，对齐 ShareLink）
- `expiresAt` 软过期，超期返回 410，DB 记录不删
- 招聘官随时可撤销（status → `revoked`）；admin 可强制重置 token

### 10.2 公开端口防护

- 公开接口不返回内部 user.id、createdBy、revokedBy 等
- URL **绝不**含候选人姓名 / 岗位（仅 token）
- 速率限制：单 token 60 秒 ≤ 30 次请求（防爆破 token）
- 公式注入防护：见 §9.2
- 输入长度限制：纪要 ≤ 500 字符；评分备注 ≤ 200 字符；候选人字段 ≤ 200 字符
- 提交后默认锁链接（按【决策点 2】决定例外）

### 10.3 数据安全

- DB 中存的内容默认明文（与现有 Review 一致）；如需加密敏感字段，复用 SystemSetting 的 AES-256-GCM 工具
- 服务端日志不打印 evaluation 完整内容，只打 evaluationId + status 变更
- R2 不参与（模板文件入仓，导出文件流式不落盘）

### 10.4 权限矩阵

| 操作 | ADMIN | RECRUITER（创建人） | RECRUITER（非创建人） | 面试官 token |
| --- | --- | --- | --- | --- |
| 列表查看 | ✅ 全部 | ✅ 自己创建 | ❌ | — |
| 创建 | ✅ | ✅ | ✅ |  — |
| 查看详情 | ✅ | ✅ 自己创建 | ❌ | ✅（仅自己 token） |
| 退回编辑 | ✅ | ❌ | ❌ | — |
| 撤销 | ✅ | ✅ 自己创建 | ❌ | — |
| 软删除 | ✅ | ❌ | ❌ | — |
| 导出 xlsx | ✅ | ✅ 自己创建 | ❌ | 按【决策点 4】 |

## 11. 校验规则

### 11.1 候选人信息

| 字段 | 创建时（招聘官） | 提交时（面试官） |
| --- | --- | --- |
| `candidateName` | 必填 | 必填 |
| `position` | 选填 | 必填 |
| `interviewDate` | 选填（默认关联 interview.scheduledAt） | 必填 |
| `interviewer` | 必填 | 必填 |
| 其它 4 个 | 选填 | 选填（但提示填写） |

### 11.2 评分

- 评分必须为 1-10 的整数（与 Excel `E10:E16` 数据校验一致）
- 提交时 7 项评分**全部必填**，缺任一项返回 `eval_validation_failed`
- 评分 ≤ 6 分时强提示填写备注（不阻止提交）

### 11.3 纪要

- `finalOpinion` 必填
- 其它 3 项选填（前端做友好提醒「未填 X 字段，是否继续提交？」）

### 11.4 计算

- `weighted[i] = round(weight[i] * score[i] / 10, 1)`
- `total = round(sum(weighted), 1)`
- `recommendation`：`total>=85`「建议录用」/ `>=75`「建议复试」/ `>=60`「谨慎考虑」/ `<60`「不建议录用」

### 11.5 导出

- 仅 `status === "submitted"` 时招聘官端常规导出按钮可用
- admin 可在 draft 状态强制导出（出现 watermark 提示「草稿」可选）

### 13.5 可访问性

- 表单控件 label 关联（`htmlFor`）

## 16. 风险与依赖

### 16.1 新增依赖

- `exceljs ^4.4.x`（后端，~5MB；纯 JS 无原生依赖，docker 镜像影响小）
- 前端无新增依赖（二维码复用现有 `qrcode.react`）

### 16.2 模板文件入仓

- `server/assets/templates/interview-evaluation-v1.xlsx` ~10KB
- 必须确认无敏感样例数据（已检查模板：仅评分标准 + 占位文字，无真实姓名）
- 入仓后通过 `git lfs` 或正常 commit 均可（10KB 量级，正常 commit）

### 16.3 部署影响

- DB migration：新增 1 张表 + 在 `candidates` / `interviews` / `jobs` 加反向关系（Prisma 自动生成 SQL，无 ALTER 风险）
- 服务重启：常规 `docker compose up -d --force-recreate backend`，无 .env 变化
- Cloudflare 长任务：导出在 3 秒内完成（10KB 模板 + 7 个评分写入），远低于 100s 上限，无需异步任务化

### 16.4 已知风险

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| ExcelJS load/save 改变模板视觉 | 导出文件与原模板不一致 | P0 强制做 load→save 回归测试，发现差异立即定位 |
| 老 WPS / 在线 Excel 打开公式不重算 | F17/G17 显示空 | DB 存 totalScore/recommendation 做兜底，后台报表不依赖 xlsx |
| 公开端口被恶意爆破 | DB 写压 | 速率限制（单 token ≤30/min） + Cloudflare WAF |
| 模板版本升级（V2 改维度/权重） | 旧评价数据语义漂移 | `templateVersion` + `templateFileHash` 落档，迁移时按版本反查权重 |

### 16.5 V2+ 待扩展

- 多面试官评分自动合并（平均分 / 加权 / 投票）
- 多轮面试横向对比（一面 vs 二面 vs HR 面）
- 评分维度 / 权重 admin 可配置（动态模板）
- 候选人岗位对比专用版（除标准单人版外，新增高管版 / 校招版 / 销售版）
- 评价完成后联动 Interview.status 自动 → 「已完成」
- 导出 PDF 版本（除 xlsx 外）
